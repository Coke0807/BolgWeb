深入理解 Kubernetes Pod 调度原理

B
博客作者
• 2026年3月15日 • 12 分钟阅读 • 2,340 次阅读
☸️

前言

Kubernetes（k8s）是目前最流行的容器编排系统。其核心功能之一就是调度（Scheduling）—— 决定将 Pod 部署到哪个 Node 上运行。理解调度原理，有助于我们在生产环境中更精细地管控资源、提升集群利用率。

调度器架构概览

kube-scheduler 是 Kubernetes 的默认调度器，以独立进程方式运行，通过 Watch API Server 感知 未绑定（Unscheduled）的 Pod，并将其与合适的 Node 进行绑定。

整个调度流程分为三个主要阶段：

  1. 预选（Filtering）：过滤掉不满足 Pod 约束条件的 Node。
  2. 优选（Scoring）：对通过预选的 Node 打分，选出最优节点。
  3. 绑定（Binding）：将 Pod 绑定到得分最高的 Node 上。
💡 从 Kubernetes 1.19 开始，调度器引入了 调度框架（Scheduling Framework）， 提供了更灵活的插件化扩展点，允许用户自定义调度逻辑。

预选阶段（Filtering）

预选阶段会对所有可用 Node 逐一执行一系列过滤插件。常见的过滤条件包括：

  • NodeResourcesFit：节点剩余资源是否满足 Pod 的 requests。
  • NodeSelector：节点标签是否匹配 Pod 的 nodeSelector。
  • TaintToleration：节点污点（Taint）是否被 Pod 容忍（Toleration）。
  • PodAffinity / PodAntiAffinity：亲和性与反亲和性规则。

优选阶段（Scoring）

通过预选的 Node 进入优选阶段。调度器对每个 Node 运行打分插件，最终得分为各插件分值的加权和。

常见打分插件：

  • LeastAllocated：优先选择资源使用率低的节点，均衡负载。
  • NodeAffinity：匹配节点亲和性规则，越匹配分值越高。
  • ImageLocality：节点上已存在所需镜像则加分，减少拉取延迟。

示例：为 Pod 设置节点亲和性

apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: kubernetes.io/arch
            operator: In
            values:
            - amd64
  containers:
  - name: app
    image: nginx:1.25
    resources:
      requests:
        cpu: "250m"
        memory: "128Mi"
      limits:
        cpu: "500m"
        memory: "256Mi"

自定义调度器插件

借助调度框架，开发者可以实现自定义插件并注册到调度器中。下面是一个用 Go 编写的最简单的 Filter 插件示例： 

package myplugin

import (
    "context"
    v1 "k8s.io/api/core/v1"
    "k8s.io/kubernetes/pkg/scheduler/framework"
)

type MyPlugin struct{}

const Name = "MyPlugin"

func (p *MyPlugin) Name() string { return Name }

func (p *MyPlugin) Filter(
    ctx context.Context,
    state *framework.CycleState,
    pod *v1.Pod,
    nodeInfo *framework.NodeInfo,
) *framework.Status {
    // 自定义过滤逻辑：跳过标记为 maintenance 的节点
    if nodeInfo.Node().Labels["status"] == "maintenance" {
        return framework.NewStatus(framework.Unschedulable, "node is under maintenance")
    }
    return nil
}

常见调度问题排查

当 Pod 长时间处于 Pending 状态时，可通过以下命令排查：

# 查看 Pod 事件
kubectl describe pod <pod-name> -n <namespace>

# 查看调度器日志
kubectl logs -n kube-system \
  $(kubectl get pods -n kube-system -l component=kube-scheduler -o name) \
  --tail=50

常见原因包括：资源不足、亲和性规则过于严格、污点未被容忍等。

总结

Kubernetes 调度器通过预选→优选→绑定三阶段为每个 Pod 找到最合适的 Node。 理解其中的插件机制，不仅有助于日常排查问题，也能让我们根据业务需求编写自定义调度策略，实现更精细的资源管控。
